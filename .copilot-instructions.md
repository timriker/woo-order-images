# AI Development Instructions for WOO Order Images

## Project Overview
WOO Order Images (WOI) is a WooCommerce plugin for managing customer-uploaded product photos. The plugin handles cropping, rotation, watermarking, admin re-cropping, and order-level print-sheet generation onto configurable paper sizes.

## Core Principles

### 0. Image Persistence Strategy
- Save each uploaded source image to disk **once**
- Do **not** create a new file on every crop or edit
- Store crop geometry (`x`, `y`, `width`, `height`) as metadata in cart and order records
- Apply crop geometry at render time for previews and print output, including puzzle splitting
- Never add synthetic white borders to persisted source files

### 1. Order-Level, Not Per-Item
- Print sheets are always **order-wide consolidation**
- Main print URL: `admin-post.php?action=woi_print_sheet&order_id=<id>`
- Rationale: reduce paper waste, reduce print jobs, simplify workflow

### 1a. Print Images Are Separate Endpoints
- Print sheets should reference per-image URLs, not inline base64 tile blobs
- Print image URL: `admin-post.php?action=woi_print_image&order_id=<id>&item_id=<id>&image_index=<n>&v=<hash>`
- The `v` key should change whenever crop, spec, or puzzle tile inputs change

### 2. Minimal, Clean HTML
- Avoid noisy `data-*` attributes for runtime-computed geometry
- Inline critical styles only when they are truly page-specific
- Keep metadata in WordPress options and post meta, not hidden DOM blobs

### 3. Customizable Over Hard-Coded
- Print margins come from `WOI_Settings`
- Bleed is a global plugin setting
- Product specs come from the product metabox
- Watermark sizing should scale from the current bleed/wrap geometry

### 4. Precise Print Production
- **Page bounds**: use explicit page dimensions and `overflow: hidden`
- **Page breaks**: only on non-last pages
- **Watermark bands**: keep them readable over future image-filled bleed areas
- **Bleed rendering**: prefer real source-image pixels for bleed where available; only stretch edges when the crop touches the source boundary

### 5. Image Orientation Handling
- Compare image aspect ratio to the product visible-area aspect ratio
- If they mismatch, swap template orientation when appropriate
- Puzzle products may also swap grid orientation (`3x4` vs `4x3`)

## Code Patterns

### Settings & Configuration
```php
$watermark_text = WOI_Settings::get_watermark_text();
$margin_top     = WOI_Settings::get_print_margin_top();
$print_bleed    = WOI_Settings::get_print_bleed();
```

### Print Rendering
- Collect images from all order items
- Read structured image entries (`url` + `crop`) from order item meta
- Resolve specs per item
- Auto-orient if needed
- Bin-pack tiles across pages
- Render absolute-positioned tiles with print-image URLs, visible windows, watermark bands, and cut outlines

### Print Image Caching
- Use versioned print-image URLs with a deterministic hash derived from source URL, crop, tile position, and print spec
- Safe browser caching is preferred when the URL changes after crop/spec updates

### Source-Pixel Bleed
- Use surrounding real source pixels for bleed whenever they exist
- Only stretch edge pixels when the crop already reaches the source-image boundary

## Common Development Tasks

### Add a New Print Setting
1. Add the constant and default in `WOI_Settings`
2. Register the setting
3. Add the field renderer to the settings page
4. Add a getter
5. Use the getter in print or frontend rendering

### Change Watermark Behavior
- Edit the band geometry in `render_print_page()`
- Search for `$top_safe`, `$bottom_safe`, `$left_safe`, and `$right_safe`
- Coordinate style updates with `assets/css/print-sheet.css`

### Adjust Tile Packing
- Modify the row/page packing closures in `render_print_page()`
- Respect page bounds and configured gap values

### Add Product-Level Setting
1. Add the field in `WOI_Admin_Product_Settings`
2. Save it with product meta
3. Include it in item/spec resolution
4. Use it in frontend and/or print rendering

### Debugging Orders in Admin
- Prefer REST, CLI, and direct print URLs before opening WooCommerce order edit screens
- It is fine to use the `AI` admin account when needed
- If you open an order in wp-admin, do not leave the order locked/open when you are done

## Testing Checklist

- [ ] Mixed product sizes (2×2, 2.5×3.5, etc.)
- [ ] Mixed image orientations (landscape, portrait, square)
- [ ] Puzzle orientation swap (`3x4` vs `4x3`)
- [ ] Multi-page packing without blank pages
- [ ] Watermark visibility and sizing
- [ ] Print margins respected in output
- [ ] Cart add/update/edit flow
- [ ] Image crop and rotation
- [ ] Admin re-crop flow
- [ ] Order cleanup behavior

## Performance Considerations

- Keep print-page HTML light by using image endpoints rather than embedding image blobs
- Prefer image ratio probes that do not require full decode when only orientation is needed
- Safe browser caching is preferred when image URLs carry version keys
- Avoid unnecessary loopback/self-HTTP requests

## Git Workflow

- Keep commits atomic and logically grouped
- Update `CHANGELOG.md` for tagged releases and notable behavior changes
- Before tagging a release, sync `readme.txt` metadata:
  - `Stable tag` should match the plugin version
  - `Tested up to` should match the WordPress version used for release testing
- The repo currently uses version commits like `v0.6.5` for release bumps; follow the existing repo convention unless explicitly asked otherwise
